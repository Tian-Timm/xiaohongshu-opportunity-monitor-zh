# 小红书机会提醒

面向不了解 Agent 的普通用户。用户只需要填写搜索需求、确认设置、阅读每日结果，并告诉 Agent 哪些值得看。

## 如何触发

支持三种方式：

```text
/小红书机会
/xhs-monitor
使用 $xiaohongshu-opportunity-monitor-zh，帮我建立小红书机会提醒。
```

下面这类自然语言也应触发：

```text
每天帮我找小红书上新发布的 AI 产品实习。
帮我盯一下上海的宠物行业招聘，不要重复帖子。
```

`$xiaohongshu-opportunity-monitor-zh` 是 Skill 可移植、可靠的明确调用方式。`/小红书机会` 和 `/xhs-monitor` 是便于记忆的文本快捷指令：宿主把它们作为消息交给 Agent 时必须触发本 Skill，但仅安装 Skill 不代表宿主界面会自动注册或补全真正的斜杠菜单命令。不得向用户声称已经注册，除非该宿主另有命令或插件机制并已验证成功。

## 触发后先让用户填写

不要一收到请求就自行确定全部搜索条件，也不要立刻开始搜索。先从用户原话中预填已知信息，再发出下面的表。告诉用户：知道的就填，不确定的可以留空，默认项可以直接保留。

| 需要的信息 | 用户填写 |
|---|---|
| 想找什么机会（必填） | 例如：AI 产品实习、独立游戏发行合作 |
| 地点或远程要求 | 例如：北京或远程；不限可留空 |
| 必须满足的条件 | 例如：日常实习、每周至少 4 天 |
| 不想看到的内容 | 例如：培训、付费内推、销售岗 |
| 发布时间范围 | 默认：最近 14 天 |
| 汇总时间 | 默认：每天 21:00（用户当地时间） |
| 已想到的关键词 | 可选；没有就由 Agent 生成 |

用户可以复制表格填写，也可以逐条用自然语言回答。目标机会仍不明确时再追问；可使用默认值的字段不要反复追问。

收到回答后，Agent 才生成搜索方案并展示确认卡：

```text
我准备这样帮你找：

目标：AI 产品、AI 运营和相近实习
地点：北京或远程
必须满足：实习或明确招继任
排除：培训、付费内推、超过两周的帖子
时间：每天 21:00 汇总
代表搜索词：AI 产品实习、智能体产品实习、组内直招、招继任……

回复“开始”即可；也可以直接告诉我要改哪一项。
```

只有用户确认后，才进行首次扫描并创建定时任务。最后明确告诉用户：首次扫描是否成功、定时任务是否确实创建成功、下一次预计运行时间。

## 必须使用真实的小红书页面

这个 Skill 不是普通网页搜索提示词。每次扫描先做**浏览器预检**，再进入网页或桌面分支：

1. 在 Codex 桌面应用中扫描小红书网站时，必须首先调用并完整阅读 `browser:control-in-app-browser`，使用侧边栏内置浏览器。它是 Codex bundled 能力；当前任务已经提供它时直接使用，不要求用户另外安装 Chrome Plugin 或 Computer Use plugin。
2. 浏览器预检完成条件是：`browser:control-in-app-browser` 已打开或读取真实的小红书页面，并能从页面读取当前 URL 和 DOM。全部满足后才开始搜索。
3. 当前任务没有提供 `browser:control-in-app-browser` 时，本轮在预检阶段结束，提示用户前往 **Settings > Browser** 启用内置 Browser，并新建任务后重试。小红书网站扫描不自动改用 Computer Use、Chrome 或通用网络搜索。
4. 只有用户明确要求操作小红书客户端或其他桌面窗口时，才调用并完整阅读 `computer-use:computer-use`。桌面预检完成条件是：能够可靠识别目标应用、当前页面和网址；任一条件无法确认时停止并请用户接管。
5. 通用网络搜索只用于查找公开资料，不算小红书站内扫描。扫描完成必须以实际操作小红书页面、执行站内搜索并读取结果为依据。
6. 第一次由用户在场运行，确认能够登录、切换搜索、读取结果和打开详情后，再建立每日自动运行。
7. 遇到登录、验证码或风控页面时暂停，让用户接管。密码和验证码始终由用户本人处理，Agent 不读取、不记录、不代填。
8. 从搜索结果卡片读取链接时，必须保留页面提供的完整 `href`，包括 `xsec_token`、`xsec_source` 等查询参数。使用页面原始 `href`，而不是只取帖子 ID 后拼成 `/search_result/<note-id>`；后者会被小红书以 `300017` 拒绝。

如果当前 Codex 没有定时任务能力，可以完成一次扫描，但必须说明无法创建每日提醒。

## Agent 如何生成搜索词

不要要求用户自己穷举关键词。根据设置表动态生成：

- 用户目标的岗位、行业、技术和机会类型；
- 招聘、HC、组内直招、招继任、急招、内推等意图；
- 地点、远程、出勤、毕业年份等条件；
- 英文缩写、公司代称、谐音和规避审核的表达；
- 仍在用户目标范围内的相近机会。

每天使用 3 个核心搜索和 2 个轮换搜索。首次运行覆盖全部词组；后续根据实际命中减少高度重复的词。向用户只展示少量代表词，不倾倒完整内部词表。

用户可随时说“也加上用户研究实习”“地点改成上海”“不要销售岗”。只有这种明确修改搜索范围的话才更新搜索设置；给单个帖子做标记不得偷偷改变关键词。

## 每天会收到什么

每天只汇报从未出现过、且未明确超过发布时间范围的新帖子。使用两个清楚的标签。

### ✅ 已确认

正文足以确认它正在提供相关机会。列出可点击原帖、公司或团队、地点、发布时间、关键要求和投递方式。

### 👀 请你核实

帖子可能有价值，但 Agent 无法低成本确认。必须写明核实原因，例如：

- 关键信息主要在图片或截图里；
- 公司使用代称、谐音或缩写；
- 标题疑似招聘，但正文信息不足；
- 发布时间或匹配程度无法确认。

不要因为无法读图就删除。只有明确是面经、求职抱怨、课程推广、培训广告、付费内推或纯经验分享，且没有招聘可能时，才直接排除。

历史重复帖、超过时间范围的帖子和明确无关内容不逐条展示，只计入末尾统计。

## 每日报告格式

```text
今晚发现 3 条新机会

[打开今日审核页](http://127.0.0.1:<port>/monitors/<monitor-id>/review)

✅ 已确认｜百度 AI 产品实习
北京｜1 天前｜每周 5 天，至少 5 个月
重点：Agent 产品、产品评测、数据分析
[打开原帖]

👀 请你核实｜“铁厂”产品实习
2 天前
核实原因：公司用了代称，具体要求在图片里。
[打开原帖]

扫描：5 组搜索 · 80 条结果 · 重复 12 条 · 超过范围 18 条

请回复我：哪些值得看、哪些你不感兴趣、哪些想稍后再看。
你可以直接说帖子标题、报告中的顺序或贴上链接。
```

不要用“值得看 1；不感兴趣 2；稍后看 3”这种容易被误解为标签编号的表达。

报告中的“打开原帖”只能使用 `ledger.py ingest` 或 `ledger.py list` 输出的 `url` 字段。不得使用账本内部的 `canonical_url`，不得删除 `url` 中的查询参数，也不得根据 `note_id` 重建链接。

没有新增时说“今晚没有发现新的候选帖子”，并给出扫描统计。登录或浏览失败时必须说明失败，不能写成“没有新增”。

## 用户标记

用户可以自然回复：

```text
百度那条值得看；“铁厂”那条我不感兴趣；剩下一条稍后看。
```

内部保存为“值得看”“不感兴趣”“稍后看”或“尚未标记”。用户隔了多天才回复、且指代可能有歧义时，先确认具体帖子。

所有出现过的帖子，无论用户如何标记，都不得再次进入“今日新增”。用户可以随时要求查看某一标记下的历史记录。

## 内部执行规则

以下细节对普通用户隐藏，但不得省略：

1. 使用 `scripts/ledger.py` 和 `scripts/daily_scan.py`，状态保存在 `work/xiaohongshu-monitor/<monitor-id>/state.json`。
2. 用 `ledger.py init` 建账本，用 `ledger.py configure` 保存目标和动态关键词；每天完成站内扫描并写好输入 JSON 后，必须用 `daily_scan.py --state <state.json> --input <scan.json>` 完成去重、写入并启动或复用本地审核页。继续用 `ledger.py mark/list/stats` 管理自然语言标记与统计。
3. 搜索结果页每组读取约 20 条，只提取帖子 ID、标题、作者、时间、完整原始链接和来源搜索，不把整页内容送入上下文。原始链接必须直接读取页面 `href`，保留全部查询参数。
4. 先做发布时间过滤，再按帖子 ID、仅供内部使用的无参数链接、作者/标题/公司/岗位指纹永久去重。所有见过的帖子都写入账本。无参数链接只能用于去重，不能展示给用户。
5. 默认淘汰运行时 14 天以前的帖子，并以 `expired_by_age` 记入账本；日期不明则进入“请你核实”。不要为补数量翻查更早帖子。
6. 只打开真正未见过的候选。每次最多详细核实约 10–15 条；其余有相关信号的候选进入“请你核实”。
7. 不复查历史岗位变化，不读取全部评论，不对所有图片批量 OCR。只有很容易看到的作者置顶停招信息才可用于判断新帖。
8. 用户标记仅供个人整理，不自动改变搜索范围。
9. 定时监测保持只读。未经明确授权，不得私信、申请、上传简历、填写表单或发送个人信息。
10. `daily_scan.py` 如果报告 `/search_result/` 链接缺少 `xsec_token`，本轮扫描视为失败：回到结果页重新读取完整 `href` 后重新执行 `daily_scan.py`。不得绕过检查、不得先更新历史状态、不得发送不可点击的报告。
11. 只有 `daily_scan.py` 返回 `ok: true` 后，才在每日报告中使用其 `review_url` 输出可点击的“打开今日审核页”。该入口会启动或复用仅监听本机的审核服务，用户不需要手动运行命令。不得自行拼接端口、监测任务 ID 或审核链接。
12. `daily_scan.py` 返回 `ok: false` 时，按其 `message` 明确报告扫描或审核页启动失败；不得声称“没有新增”，也不得提供审核链接。扫描失败时不得把失败结果当成空审核队列。

保持内部流程严谨，但始终用普通人的语言与用户沟通。
